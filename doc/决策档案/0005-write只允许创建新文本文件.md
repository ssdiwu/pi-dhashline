# ADR 0005：write 只允许创建新文本文件

## 状态

Accepted

## 背景

Pi 原生 `write` 同时支持创建与覆盖，模型因此可以绕过 DHashline 的文件标签、已见行和 stale 校验直接重写已有文件。完全禁止 `write` 会失去创建文件能力；给整文件覆盖增加标签则会与精确 `edit` 重叠，并鼓励高成本 whole-file rewrite（整文件重写）。

## 决策

- DHashline 接管同名 `write`，保留原生 `path + content` 参数形状，但只允许创建不存在的 UTF-8 文本文件。
- 在已存在且非符号链接的父目录中先准备并验证完整临时文件，再以 hard-link create 原子发布目标；目标已存在、是符号链接或在竞态中出现时失败关闭，不能覆盖任何字节。
- 不自动创建父目录，避免目录创建与失败清理扩大路径竞态；调用方必须先明确创建目录。
- 成功后记录 fresh snapshot 与 tag，但不授予 seen-lines；后续编辑必须重新 `read` 或 `search`。
- `write` 使用与其他公开工具相同的两层人类可读投影，展开态只显示创建元数据，不回显完整正文。

## 权衡

- 优点：关闭最直接的安全绕过，保持创建文件路径和原生参数形状，错误不会破坏已有文件。
- 代价：依赖原生 `write` 覆盖文件的任务需要改用 `read` + `edit`；`bash` 仍可显式修改文件，且 Node 标准库不提供目录句柄式 `openat`，因此本契约防止正常并发下的目标覆盖，不把同一系统用户恶意替换路径命名空间视为可防御沙箱边界。
